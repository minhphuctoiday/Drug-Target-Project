Product documentation

Protein target discovery docs for the LUAD Big Data platform.

This documentation explains how the platform integrates TCGA/GDC, GEO and STRING artifacts to rank candidate protein targets for lung adenocarcinoma, then serves interactive biological evidence.

What This Platform Does

The platform is a Big Data analytics workspace for drug target identification. It consumes refined Parquet artifacts generated by the DE pipeline, computes DA/ML evidence, and exposes protein target evidence through FastAPI and a research dashboard.

Integrates omics datasetsTCGA/GDC expression, GEO validation and STRING PPI network.
Ranks protein targetsCombines expression, validation, STRING graph, druggability and unsupervised ML priority.
Separates model rolesProtein target ranker is primary; expression classifier is supporting evidence only.
Visualizes biological networksVolcano, heatmap, PPI network, enrichment and model evidence views.

End-to-End Workflow

  1. Data Engineering

    Raw TCGA/GDC, GEO and STRING data are cleaned into refined Parquet files.

  2. Data Analysis

    Computes differential expression, FDR, GEO signal, STRING PPI metrics, PCA/KMeans and target score.

  3. Biological Evidence

    Adds druggability, curated pathway enrichment, exploratory survival association and protein target reports.

  4. Primary Machine Learning

    Trains an unsupervised protein target ranker using Isolation Forest, Gaussian Mixture rarity and KMeans cluster priority.

  5. Supporting Machine Learning

    Keeps a regularized expression classifier only for auxiliary tumor-like phenotype evidence.

  6. Serving + Visualization

    FastAPI serves target ranking, visualizations, reports, model evidence and chatbot responses.

Dashboard Reading Order

The dashboard is ordered to match the project objective. Start with the Big Data evidence layer, then inspect target ranking, biological context, PPI network structure and model evidence. The expression probe appears later because it is a supporting model, not the primary target-discovery system.

1. OverviewScale of targets, PPI edges, validation signal and current top target.
2. Data FabricHow TCGA/GDC, GEO, STRING, DA artifacts and serving layers connect.
3. Targets + VolcanoMain protein target ranking beside differential-expression evidence.
4. Pathways + EvidenceFunctional programs and score decomposition for prioritized targets.
5. Heatmap + PPIExpression patterns and STRING interaction network for top targets.
6. Model EvidencePrimary unsupervised ranker plus secondary expression classifier.

Target Score vs Integrated Evidence

The platform intentionally separates ranking concepts. Target score is the DA-driven target prioritization score. Integrated evidence score adds druggability, survival-time association and model importance to make a richer protein target report. 

target_score =
  expression evidence
+ PPI network context gated by DE strength
+ GEO validation
+ sample prevalence
+ biological penalties
integrated_evidence_score =
  0.35 * expression_component
+ 0.20 * network_component
+ 0.17 * validation_component
+ 0.14 * druggability_score
+ 0.08 * model_component
+ 0.06 * survival_component
protein_ml_priority_score =
  0.34 * IsolationForest priority
+ 0.24 * GaussianMixture rarity
+ 0.24 * KMeans cluster priority
+ 0.18 * biological evidence prior

ML Models

Primary: Protein Target Ranker

The primary ML model is unsupervised because the project does not have reliable labels such as "true drug target" versus "not a target". It ranks candidate protein targets from DA, STRING PPI and evidence features, then outputs ML priority and target-cluster fields.

TaskUnsupervised candidate protein target prioritization.
AlgorithmsIsolation Forest, Gaussian Mixture rarity and KMeans clustering.
InputsExpression evidence, FDR, target score, STRING degree/PageRank/betweenness, GEO signal and auxiliary model importance.
Outputsprotein_ml_priority_score, protein_ml_rank, protein_target_cluster.
Why no confusion matrix?There are no target/non-target labels, so supervised classification metrics would be misleading.

Supporting: Expression Classifier

The classifier receives protein-coding gene expression values and estimates whether a profile looks more like TCGA-LUAD tumor or adjacent normal tissue. This is not the main task. It is retained as an explainability layer: if a protein-coding feature helps separate tumor-like expression, its contribution can support, but not determine, the protein target ranking.

The model is a supporting research signal, not a clinical diagnostic device and not the primary target-discovery output.
Best modelLogistic Regression L2
Selection protocolModel and threshold selected on validation split.
Strict test protocolTrain-only model evaluated once on held-out test set.
Why not deep learning?Samples are small relative to gene features; regularized linear models are safer and more explainable.

Chatbot

The chatbot is grounded in an English Markdown project knowledge base at docs/project_knowledge_base.md. The backend chunks this file, builds a cached vector index, retrieves relevant chunks, injects live artifact evidence for mentioned targets, and uses Gemini to answer project-specific questions.

Generation modelgemini-2.5-flash
Embedding modelgemini-embedding-001
Index cacheoutputs/rag/project_knowledge_index.json
Artifact groundingQuestions mentioning PLK1, SPP1, TOP2A and other targets also retrieve rank, score, FDR, GEO and STRING metrics from local artifacts.
LanguageQuestions may be entered by the user, but answers and retrieved documentation are maintained in English.
SecurityPrompt-injection checks block requests to reveal hidden prompts, ignore instructions, bypass safeguards or expose secrets.
Retrieved chunks are treated as untrusted context. The assistant must use them as evidence, not as instructions.

Protein Target Report

Each protein target report combines DA, ML and biological annotations into one exportable view. The report includes target rank, integrated score, expression evidence, GEO validation, drug class, known drugs, pathway context, survival-time association and model importance. The URL still uses a gene symbol because expression data is keyed by the protein-coding gene that encodes the candidate target.

Example endpoint:

GET /api/gene/PLK1/report

Scientific Limitations

  • External tumor-normal validation is not added because project input data is frozen.
  • GEO is used as stage-association validation, not as direct tumor-normal validation.
  • Survival evidence is exploratory because current refined labels do not include reliable censor/event metadata.
  • Druggability annotations are curated local knowledge, not a full DrugBank/Open Targets integration.
  • Classifier features are not automatically protein drug targets; target evidence must combine biological context.
  • The platform supports research prioritization and education, not clinical diagnosis.
